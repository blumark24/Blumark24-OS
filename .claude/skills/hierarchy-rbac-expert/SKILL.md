# hierarchy-rbac-expert

## الهدف
ضمان سلامة الهيكل الإداري والموظفين والأدوار والصلاحيات في Blumark24 OS وربطها بالمنشأة بطريقة واضحة وآمنة.

## متى تستخدمها
- عند تعديل الهيكل الإداري أو الموظفين أو الأدوار.
- عند إضافة permission gate أو مراجعة وصول وحدة تشغيلية.
- عند ربط المكتب الافتراضي أو AI بالهيكل والصلاحيات.

## حدود الصلاحية
- لا تعدل Supabase/RLS/migrations إلا بطلب صريح.
- لا تستخدم بيانات وهمية لموظفين أو أدوار أو وحدات إدارية.
- لا تغير التصميم أو RTL أو الهوية دون طلب.
- لا تستخدم localStorage للبيانات التجارية أو الحساسة.
- اجعل التعديل صغيرًا ومحدود النطاق.

## قواعد Blumark24 OS الخاصة
- التدفق المعتمد: Owner -> Organization -> Settings -> Administrative Structure -> Employees -> Roles & Permissions -> Operational Modules -> Virtual Office -> AI.
- الهيكل الإداري يسبق الموظفين والصلاحيات والوحدات التشغيلية.
- كل صلاحية يجب أن ترتبط بالمنشأة والدور والسياق.
- احترم الباقات عند تفعيل أو حجب قدرات.
- اللغة العربية والسوق السعودي أولًا.
- احم RTL والهوية عند عرض الهياكل والأدوار.

## قائمة فحص قبل التعديل
- حدد المستوى المتأثر: منظمة، إدارة، قسم، فريق، موظف، دور، صلاحية.
- تحقق من tenant isolation.
- راجع مصدر بيانات الموظفين والأدوار.
- تأكد من عدم تجاوز الباقات.
- تأكد من عدم الحاجة لتعديل قاعدة البيانات دون تصريح.

## قائمة فحص بعد التعديل
- تحقق أن الوصول يعمل حسب الدور والصلاحية.
- شغل lint/build عند تعديل كود.
- راجع الحالات الفارغة بدون بيانات وهمية.
- تأكد أن الهيكل لا يعرض بيانات منشأة أخرى.
- وثق أثر التعديل على الوحدات التشغيلية أو المكتب الافتراضي.

- طريقة ربطها بالمنشأة والباقات.
- نتيجة lint/build.
- أي مخاطر صلاحيات متبقية.
