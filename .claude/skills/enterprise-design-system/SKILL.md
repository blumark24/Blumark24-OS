# enterprise-design-system

## الهدف
حماية التصميم المؤسسي لـ Blumark24 OS وضمان عدم كسر الواجهة، RTL، الهوية البصرية، المكونات، أو تجربة الاستخدام العربية.

## متى تستخدمها
- عند تعديل واجهة أو مكون React/TSX.
- عند مراجعة ألوان، خطوط، تخطيط، spacing، حالات فارغة، أزرار، جداول، أو بطاقات.
- عند التأكد أن التصميم بقي مناسبًا لسوق سعودي وعربي أولًا.

## حدود الصلاحية
- ممنوع تغيير التصميم بدون طلب صريح.
- ممنوع تغيير الهوية أو RTL أو اتجاه النص بلا ضرورة موثقة.
- ممنوع إدخال بيانات وهمية داخل الواجهة.
- ممنوع تعديل Supabase أو RLS أو migrations بدون طلب صريح.
- أي تعديل يجب أن يكون صغيرًا ومحدود النطاق.

## قواعد Blumark24 OS الخاصة
- النظام يعمل بتدفق: Owner -> Organization -> Settings -> Administrative Structure -> Employees -> Roles & Permissions -> Operational Modules -> Virtual Office -> AI.
- اللغة العربية أولًا، والنصوص يجب أن تكون واضحة واحترافية.
- يجب حماية الهوية البصرية الحالية بدل إعادة تصميمها.
- الحالات الفارغة يجب أن تكون صادقة ولا تعرض أرقامًا أو نشاطًا غير حقيقي.
- اربط أي تحكم واجهي بالمنشأة والصلاحيات والباقات عند الحاجة.
- لا تستخدم localStorage للبيانات التجارية أو الحساسة.

## قائمة فحص قبل التعديل
- حدد هل المطلوب تعديل تصميم أم إصلاح سلوك فقط.
- افحص المكونات القريبة لتقليد النمط الحالي.
- تأكد من عدم تغيير الألوان أو التخطيط أو RTL دون طلب.
- تأكد من أن النص العربي مناسب ومختصر.
- تحقق من عدم وجود fallback وهمي.

## قائمة فحص بعد التعديل
- راجع أن العناصر لا تتداخل ولا تكسر responsive layout.
- تحقق من بقاء RTL والهوية.
- شغل lint/build عند تعديل كود الواجهة.
- راجع الحالات الفارغة والتحميل والخطأ.
- تأكد أن الملفات المعدلة ضمن النطاق المطلوب.

## مخرجات التقرير المطلوبة من Claude Code
- الملفات المعدلة.
- عناصر الواجهة التي تغيرت ولماذا.
- تأكيد حماية RTL والهوية.
- نتيجة lint/build.
- أي قيود أو نقاط تحتاج مراجعة بصرية.
